<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Auth.js credentials action | Untrusted email/password values enter server auth code. |
| session cookie -> protected admin pages | Browser-held session state controls access to operational data. |
| seed env -> database | Local secret inputs create privileged owner credentials. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | `auth.ts`, `lib/auth/admin-users.ts`, `lib/auth/login-throttle.ts` | mitigate | Use Auth.js Credentials Provider, normalized email lookup, active-user requirement, generic login failure, tested scrypt password verification, and an in-process 5-failure/15-minute throttle keyed by normalized email plus source IP when available. |
| T-02-02 | Elevation of privilege | `proxy.ts`, `lib/auth/admin-session.ts`, `app/admin/page.tsx` | mitigate | Protect `/admin` with proxy and call `requireAdmin()` inside the protected page; later setup actions must also call `requireAdmin()`. |
| T-02-03 | Information disclosure | Auth callbacks and admin DTOs | mitigate | Session DTO contains only id, email, role, and active; tests assert `passwordHash` is not returned. |
| T-02-04 | Repudiation | `scripts/seed-admin-user.mjs` | accept | Phase 2 seeds one owner account from local env; no admin-user management UI exists in this phase. |
</threat_model>
