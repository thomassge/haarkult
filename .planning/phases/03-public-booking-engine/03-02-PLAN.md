<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public browser to `/api/booking/slots` | Query parameters are attacker-controlled and must be schema-validated. |
| Public slot endpoint to database setup data | Endpoint may expose only safe available slot DTOs, not internal setup diagnostics or full staff records. |
| Availability engine to booking state | Calculated slots must exclude stale, blocked, or policy-invalid windows before the browser sees them. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Tampering | `slotQuerySchema` and `getPublicAvailableSlots` | mitigate | Validate service/date/staff inputs with Zod and verify service/staff ids against catalog plus active staff-service assignments before calculating slots. |
| T-03-02-02 | Information Disclosure | `/api/booking/slots` error path | mitigate | Return public-safe German errors such as `Ungueltige Anfrage.` without database details, stack traces, admin setup labels, or internal ids beyond selected slot DTOs. |
| T-03-02-03 | Denial of Service | slot candidate generation | mitigate | Bound date queries to one selected date, enforce booking horizon from config, and step through candidates by configured `slotStepMinutes` only. |
| T-03-02-04 | Tampering | `calculateAvailableSlots` | mitigate | Treat browser staff preference as a filter only; concrete staff id on returned slots is derived from server-owned staff/service/schedule data. |
</threat_model>
