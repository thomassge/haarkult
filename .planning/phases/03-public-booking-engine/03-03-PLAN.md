<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public browser to submit Server Action | FormData contains untrusted service, staff, slot, date, and contact fields. |
| Submit action to availability foundation | Slot data can become stale between lookup and submit and must be recomputed server-side. |
| Submit action to database writes | Personal data and booking records cross into persistent storage. |
| Server result to public UI | Conflict, validation, and success messages must be German and public-safe. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering | `publicBookingSubmissionSchema` | mitigate | Validate all submitted fields with Zod and re-resolve service/staff/slot through server-owned catalog, setup, and availability data. |
| T-03-03-02 | Tampering | `createPublicBooking` | mitigate | Re-run `getPublicAvailableSlots()` during submission and insert only when the selected `slotId`, concrete `staffId`, and `startAt` are still available. |
| T-03-03-03 | Information Disclosure | validation and conflict errors | mitigate | Return German public-safe messages and never expose stack traces, database errors, admin setup state, or internal credentials. |
| T-03-03-04 | Repudiation | booking creation | mitigate | Insert an initial `bookingEvents` row with `eventType: "created"` and `actorType: "customer"` for every successful public booking. |
| T-03-03-05 | Privacy | contact form and booking insert | mitigate | Collect only required name, phone, email, and optional note; do not add accounts, payment data, birthdates, addresses, or marketing opt-ins. |
</threat_model>
