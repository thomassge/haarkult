# Milestone v1.0: Phase 1 - Builder Boundaries & Mode Hardening

**Status:** SHIPPED 2026-04-12
**Phases:** 1
**Total Plans:** 4

## Overview

Phase 1 protected the reusable salon-builder foundation before booking and admin complexity grow. It closed the architecture and mode-contract requirements needed before Phase 2 can add protected staff operations.

## Phase

### Phase 1: Builder Boundaries & Mode Hardening

**Goal:** Protect the reusable salon builder architecture before booking and admin complexity grow
**Depends on:** Nothing
**Requirements:** BUIL-01, BUIL-02, MODE-01, MODE-02, MODE-03
**UI hint:** no

**Success Criteria:**

1. Maintainer can swap salon content/config and assets without rewriting shared brochure components.
2. `contact_only` mode exposes contact paths and does not expose a working public booking flow.
3. `booking` mode keeps booking/admin boundaries explicit and separate from brochure-only components.

**Plans:**

- [x] 01-01: Normalize reusable salon content and configuration boundaries
- [x] 01-02: Harden mode gating for public booking entry points and fallback contact behavior
- [x] 01-03: Isolate shared booking/admin module boundaries from brochure components
- [x] 01-04: Close MODE-03 fallback hidden-channel drift

## Milestone Summary

### Key Accomplishments

- Split booking-domain configuration and copy into `content/booking.ts` while keeping `content/site.ts` salon-wide.
- Standardized internal booking mode vocabulary to `contact_only` and `booking`.
- Added brochure-safe selectors in `lib/site-mode.ts` for booking visibility, fallback contact resolution, and public actions.
- Added a Vitest unit/component harness for Phase 1 regression coverage.
- Isolated `/termin-buchen` route-local booking entry modules and created an explicit `/admin` boundary placeholder.
- Closed the MODE-03 fallback drift by removing static hidden-channel copy and adding a config-mutation regression test.

### Key Decisions

- Booking rules and booking-page copy belong in `content/booking.ts`; salon identity and contact details remain in `content/site.ts`.
- Brochure surfaces consume mode behavior through selector helpers rather than raw config branches.
- Public footer and homepage actions use the same resolved public-action contract.
- `/admin` exists as a route boundary before auth so Phase 2 can add protection cleanly.
- Channel-specific fallback wording is selector-driven; static fallback steps stay channel-agnostic.

### Issues Resolved

- Removed mixed `online_booking` versus `booking` vocabulary from the touched config and route surfaces.
- Removed hidden-channel drift from `/termin-buchen` contact-only fallback copy.
- Added automated tests for selector behavior, route fallback behavior, and brochure/booking import boundaries.

### Issues Deferred

- Admin authentication and protected setup flows move to Phase 2.
- Public booking engine, availability, stylist selection, and booking submission move to Phase 3.
- Staff booking operations and notifications move to Phase 4.
- Launch hardening and second-salon reuse proof move to Phase 5.

- Phase verification: passed, 7/7 must-haves verified.
- Milestone audit: passed, 5/5 scoped requirements satisfied.
- Nyquist validation: compliant.
